Q1: The TMS handles all transaction coordination like managing concurrency control, conflict detection, and ordering of transactions while the SS handles storage and serving reads. They are decoupled so each layer can scale independently. The TMS can process high volumes of concurrent transactions without being bottlenecked by disk I/O, and the SS can optimize for read/write throughput without worrying about transaction logic. This also improves fault tolerance, since a failure in one layer won't bring down the other.
Q2: The Sequencer is responsible for detecting read-write conflicts between concurrent transactions and does this by comparing the read sets of committing transactions against writes that occurred after their read version. This conflict check happens during the commit phase, after the client sends its read/write sets to the Proxies, which forward them to the Sequencer for validation before any writes are applied.
Q3: The read version defines the consistent snapshot of the database that a transaction operates on, making sure it only sees data committed before that point. This gives FDB serializable isolation, since all reads within a transaction reflect a single consistent state and the conflict check uses this version to detect any writes that occurred after the transaction started.
Q4: Snapshot reads bypass conflict detection, meaning they won't cause a transaction to abort even if another transaction writes to the same keys. I would choose snapshot reads if I needed to avoid conflicts on frequently read keys and can tolerate slightly stale data, for exaple in generating a dashboard that shows total ride counts per taxi company.
Q5: StorageServers continuously pull committed data from LogServers during normal operation, meaning the durable state is already reflected in storage without needing to replay a log on restart. When a failure occurs, the system only needs to recruit a new TMS and resume from the last committed version, since LogServers have already saved the writes and StorageServers just need to catch up on the small amount of data they missed since the last sync.
